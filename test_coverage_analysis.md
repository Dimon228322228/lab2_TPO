## Обзор тестовых классов

Проект включает модульные тесты для каждого класса модулей (ядерных и производных), написанные с использованием JUnit 5. Тесты расположены в директории `src/test/java/lab2/`.

### Список тестовых классов:

1. **CosineTest** – тестирование класса `Cosine`.
2. **NaturalLogarithmTest** – тестирование класса `NaturalLogarithm`.
3. **SineNamedTest** – тестирование класса `SineNamed`.
4. **TangentNamedTest** – тестирование класса `TangentNamed`.
5. **CotangentNamedTest** – тестирование класса `CotangentNamed`.
6. **SecantNamedTest** – тестирование класса `SecantNamed`.
7. **CosecantNamedTest** – тестирование класса `CosecantNamed`.
8. **LogarithmNamedTest** – тестирование класса `LogarithmNamed`.
9. **ComplexNamedTest** – тестирование класса `ComplexNamed`.



## Стратегия тестирования

### 1. **Параметризованные тесты**
Большинство тестов используют `@ParameterizedTest` с различными источниками данных (`@ValueSource`, `@MethodSource`). Это позволяет проверить множество входных значений без дублирования кода.

**Пример:**
```java
@ParameterizedTest
@ValueSource(doubles = {0.0, Math.PI / 6, Math.PI / 4, Math.PI / 3, Math.PI / 2, Math.PI})
void computeBasicValues(double x) {
    // вызов compute и сравнение с Math.cos(x)
}
```

**Объяснение выбора:** Параметризация увеличивает покрытие граничных значений и типичных точек (нули, экстремумы, особые углы). Это эффективно для математических функций, где важно проверить корректность на множестве аргументов.

### 2. **Тестирование точности**
В каждом тесте задаётся допустимая погрешность (`PRECISION = 1e-10`). Результаты сравниваются со значениями стандартной библиотеки `Math` (которая служит эталоном).

**Объяснение выбора:** Поскольку реализации используют ряды Тейлора, важно убедиться, что достигнута заявленная точность. Сравнение с `Math` даёт уверенность в корректности алгоритма.

### 3. **Тестирование исключительных ситуаций**
Тесты проверяют, что при недопустимых аргументах выбрасываются соответствующие исключения (например, `IllegalArgumentException` для отрицательного логарифма, `ArithmeticException` для деления на ноль).

**Пример:**
```java
@ParameterizedTest
@ValueSource(doubles = {0.0, -1.0, -5.0})
void computeNonPositiveThrows(double x) {
    assertThrows(IllegalArgumentException.class, () -> ln.compute(x));
}
```

**Объяснение выбора:** Обработка ошибок – важная часть надёжности. Тесты гарантируют, что модули ведут себя предсказуемо в ошибочных сценариях.

### 4. **Тестирование граничных значений и сингулярностей**
Для тригонометрических функций тестируются точки, где знаменатель близок к нулю (например, `π/2` для тангенса). В таких случаях либо ожидается исключение, либо большое значение (из-за численной неустойчивости).

**Объяснение выбора:** Сингулярности – типичные места ошибок. Тесты помогают убедиться, что реализация либо корректно выбрасывает исключение, либо возвращает адекватно большое значение.

### 5. **Тестирование зависимостей между модулями**
Производные модули (например, `SineNamed`) зависят от ядерных (`Cosine`). Тесты создают экземпляры зависимостей и проверяют, что вычисления используют их правильно.

**Пример:** `SineNamedTest` создаёт `Cosine` и проверяет тождество `sin(x) = cos(π/2 - x)`.

**Объяснение выбора:** Это гарантирует, что композиция модулей работает корректно и изменения в ядерных классах не сломают производные.

### 6. **Тестирование имени модуля (`getName`)**
Каждый тестовый класс включает тест для метода `getName()`, проверяющий, что имя соответствует ожидаемому (используется для формирования имён CSV‑файлов).

**Объяснение выбора:** Имя – часть контракта интерфейса `NamedOutputFunction`. Его корректность важна для корректной работы `FunctionManager`.

## Покрытие кода (оценка)

На основе просмотра тестов можно сделать выводы:

- **Высокое покрытие** для всех модулей вычисления функций: тестируются основные значения, граничные случаи, исключения, точность.
- **Среднее покрытие** для комплексной функции `ComplexNamed`: тестируются обе ветви (тригонометрическая и логарифмическая), но из‑за сложности формулы используются большие допуски.
- **Низкое покрытие** для инфраструктурных классов (`FunctionManager`, `Range`, `CSVWriter`, `App`). Возможно, они тестируются интеграционно или считаются достаточно простыми, чтобы обойтись без модульных тестов.

## Обоснование выбора тестового покрытия

Выбор именно такого тестового покрытия обусловлен следующими принципами и ограничениями:

### 1. **Приоритетность модулей вычисления**
   - **Критичность для бизнес-логики:** Основная цель проекта – корректное вычисление математических функций. Ошибки в этих модулях приведут к неверным данным в CSV‑файлах, что делает их тестирование наиболее важным.
   - **Сложность алгоритмов:** Реализации используют ряды Тейлора, итерационные методы, преобразования аргументов. Эти алгоритмы подвержены ошибкам округления, сходимости, обработки особых случаев. Поэтому тесты должны быть максимально полными.

### 2. **Экономия усилий за счёт параметризации**
   - Вместо написания десятков отдельных тестов для каждого значения используется параметризация, что позволяет охватить множество тестовых случаев при минимальном объёме кода.
   - Это соответствует принципу **DRY (Don’t Repeat Yourself)** и упрощает поддержку тестов.

### 3. **Использование эталонной реализации (Math)**
   - Стандартная библиотека Java `Math` считается эталонной, её результаты принимаются за истинные. Это позволяет избежать написания сложных эталонных вычислений вручную.
   - Такой подход **снижает риск ошибок в самих тестах** и ускоряет их разработку.

### 4. **Фокус на граничные значения и исключения**
   - Опыт показывает, что большинство ошибок возникает на границах областей определения (нули, сингулярности, очень большие/малые значения). Поэтому тесты специально включают такие точки.
   - Проверка исключений гарантирует, что модуль не «молча» возвращает некорректные значения (например, `NaN` или `Infinity`), а явно сообщает об ошибке.

### 5. **Пропуск инфраструктурных классов**
   - Классы `FunctionManager`, `Range`, `CSVWriter` и `App` рассматриваются как **инфраструктурные**, их логика относительно проста (в основном, организация данных и ввод‑вывод).
   - Решение не писать для них модульные тесты может быть связано с:
     - **Временными ограничениями** (фокус на основном функционале).
     - **Планами покрыть их интеграционными тестами** (например, тестами, которые запускают `App` и проверяют создание файлов).
     - **Уверенностью в корректности** благодаря использованию стандартных библиотек (например, `Files.newBufferedWriter`) и простой логике.
   - Однако это создаёт риск: ошибки в этих классах могут остаться незамеченными. Поэтому в рекомендациях предложено добавить тесты для них.

### 6. **Баланс между полнотой и скоростью выполнения**
   - Тесты должны выполняться быстро, чтобы не замедлять разработку. Параметризованные тесты с разумным количеством значений обеспечивают хороший баланс.
   - Для `ComplexNamed` используется больший допуск (`TOLERANCE = 1e-6`), потому что формула содержит множество операций, накапливающих погрешность. Жёсткая проверка точности привела бы к хрупким тестам.

### 7. **Следование принципам тест‑дизайна**
   - **Эквивалентное разбиение:** Значения аргументов разбиты на классы (положительные, отрицательные, нули, особые точки) – тесты покрывают каждый класс.
   - **Анализ граничных значений:** Для каждого класса проверяются границы (например, `x = 0.0` для логарифма, `x = π/2` для тангенса).
   - **Таблица принятия решений:** В тестах `ComplexNamed` явно разделены два случая (`x <= 0` и `x > 0`), что соответствует полному покрытию ветвлений.

### 8. **Обеспечение регрессионной защиты**
   - Наличие тестов для каждого модуля позволяет быстро обнаруживать регрессии при изменении кода (например, при оптимизации алгоритма или исправлении ошибок).
   - Тесты имени (`getName`) защищают от случайного изменения имени, которое сломает формирование имён CSV‑файлов.

## Заключение

Тестовое покрытие проекта сфокусировано на модулях вычисления функций и обеспечивает уверенность в их корректности. Стратегия тестирования использует параметризацию, сравнение с эталонной реализацией и проверку исключений, что соответствует лучшим практикам тестирования математических библиотек. Для полноты покрытия рекомендуется добавить тесты для инфраструктурных компонентов.
